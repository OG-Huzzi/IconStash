1. Why Accessibility Matters for Vectors

Unlike text, which screen readers parse natively, vector paths (the `` coordinate string) carry no semantic meaning to assistive technologies. Left unconfigured, a screen reader may attempt to read out the raw filename of the SVG, speak unhelpful unicode codes, or bypass critical interactive tools entirely.

To design an inclusive interface, you must classify every SVG icon into one of three semantic categories: **decorative**, **interactive**, or **informative**.

2. Decorative Icons (Hiding from Screen Readers)

The vast majority of UI icons are purely decorative. They reinforce a nearby text label (for example, a magnifying glass icon next to the word "Search"). If a screen reader reads both the icon and the text, it creates annoying repetition (e.g. "magnifying glass, search, button").

To ensure screen readers skip decorative icons, add the aria-hidden="true" attribute to the SVG tag, and ensure no nested `` or labels exist inside the markup:</p> <pre><code><!-- Decorative icon: invisible to assistive screen readers --> <svg viewBox="0 0 24 24" aria-hidden="true" focusable="false"> <path d="M12 2v20M17 5H9.5a3.5 3.5 0 0 0 0 7h5a3.5 3.5 0 0 1 0 7H6"/> </svg> <span>Budget Settings</span></code></pre> <h2 id="interactive">3. Interactive Icons (Adding Text Alternatives)</h2> <p>When an icon is the only element inside a trigger (like a close 'X' button or an arrow slide button), it is an interactive control. Without a label, the user has no way of knowing what the button does.</p> <p>In this case, place the text label directly on the **outer wrapper element** (like the button or link) using an <code>aria-label</code>, and hide the child SVG entirely:</p> <pre><code><!-- Interactive icon button --> <button class="close-btn" aria-label="Close modal dialog"> <svg viewBox="0 0 24 24" aria-hidden="true" focusable="false"> <path d="M18 6 6 18M6 6l12 12"/> </svg> </button></code></pre> <p>This layout ensures that the user hears "Close modal dialog, button" instead of the raw SVG file attributes.</p> <h2 id="standalone">4. Standalone Informative Vectors</h2> <p>If an SVG contains unique information (such as an interactive chart, diagram, or standalone illustration) and does not sit inside an interactive button, it should be treated as a semantic image. Use nested `<title>` and `<desc>` tags, and link them using `aria-labelledby`:</p> <pre><code><svg viewBox="0 0 24 24" role="img" aria-labelledby="chart-title chart-desc"> <title id="chart-title">Monthly Sales Chart</title> <desc id="chart-desc">Bar chart displaying our quarterly growth, peaking at 40% in June.</desc> <rect x="2" y="10" width="4" height="12"/> <rect x="8" y="6" width="4" height="16"/> </svg></code></pre> <h2 id="focus-tab">5. Eliminating Ghost Focus Stops</h2> <p>Internet Explorer (and some older edge/webkit engines) has a known bug where SVG elements are included in the keyboard tab index, even if their parent container is not focusable. This results in "ghost" tab stops where keyboard users have to press tab multiple times to pass through an icon.</p> <p>To prevent this, always add the <code>focusable="false"</code> attribute to every SVG tag in your project.
